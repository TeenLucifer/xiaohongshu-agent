# Python AI Starter Repository Rules

始终使用中文简体回复。

## 项目定位
- 本模板用于初始化单仓的 Python AI 应用项目。
- 默认目标是快速建立可维护、可测试、可观察的工程骨架。
- 第一阶段优先完成可运行的 feature 骨架，不预设具体业务领域。

## 工程优先级
1. 可测试性
2. 可维护性
3. 可调试性
4. 迭代速度

## 工作模式
- 所有功能开发默认走 `SDD + 严格 TDD`。
- 开始实现前，必须先阅读：
  - 根目录 `AGENTS.md`
  - 对应 `specs/<feature>/spec.md`
  - 对应 `specs/<feature>/tasks.md`
- 如果功能没有对应 spec，先补 spec，再写测试，再实现。
- 完成后必须同步更新：
  - `specs/<feature>/tasks.md`
  - `specs/<feature>/acceptance.md`

## Python 与工具链
- Python 版本固定为 `3.11+`。
- 统一使用 `uv` 管理依赖与命令执行。
- 质量门禁固定为：
  - `uv run ruff check .`
  - `uv run ruff format --check .`
  - `uv run pyright`
  - `uv run pytest -q`
- 没有通过上述检查，不算完成。

## 代码风格
- 公共函数、类方法、协议、schema 必须带完整类型标注。
- 数据边界优先使用 `pydantic` 模型；不要在系统边界上传裸 `dict`。
- 模块职责保持单一；避免巨型文件与巨型函数。
- 复杂逻辑先拆成小函数，再考虑抽象类或框架。
- 统一使用 `logging`，禁止把 `print` 留在正式逻辑中。
- 配置值不得硬编码在业务模块中；统一通过配置层读取。
- 自定义业务异常集中定义，入口层负责转换与展示。

## 架构边界
- 目录结构由具体项目自行决定，但必须保证模块职责清晰。
- 页面、用例编排、结构化类型、外部模型适配和持久化边界应显式分离。
- 不要把所有逻辑堆进单个文件或单个入口。
- 通用辅助层不允许承载核心业务规则。

## SDD 规则
- 每个 feature 固定包含：
  - `spec.md`
  - `tasks.md`
  - `acceptance.md`
- `spec.md` 定义需求与边界。
- `tasks.md` 定义先测后写的实施顺序。
- `acceptance.md` 定义手工与自动化验收路径。
- 任何公共接口变化都必须反映到 spec 与 acceptance。

## TDD 规则
- 先写失败测试，再实现最小代码，再重构。
- 每个 feature 至少覆盖：
  - 必要的单元测试
  - 关键链路的集成测试
  - 主用户流的最小 E2E
- 如果某个 UI 细节不适合先写精细测试，也必须先定义 acceptance 场景。
- 没有测试支撑的重构不允许进入主分支。

## 禁止事项
- 不要跨层直接访问内部实现细节。
- 不要把多个职责塞进单个文件或类中。
- 不要绕过 schema 在模块之间传递不透明数据。
- 不要顺手扩展 spec 外的能力。
- 不要把临时调试代码遗留在正式路径中。
