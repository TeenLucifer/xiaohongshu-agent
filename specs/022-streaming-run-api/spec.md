# 022 Streaming Run API

## 背景

`020-backend-glue-minimal` 的同步 `POST /runs` 已足够打通最小后端，但前端后续需要更好的运行中体验，包括逐步显示模型回复、工具调用与关键运行事件。流式输出需要在已有后端入口上独立定义，避免和最小胶水层混在一起。

## 目标

- 在现有后端 run 语义上补充流式运行协议
- 固定流式事件边界与最小事件类型
- 支持前端主栏渐进显示运行结果
- 与同步 `POST /runs` 保持并存，而不是替换

## 非目标

- topic 真相层
- 数据库
- 队列与复杂任务调度
- 多 worker 分发
- 浏览器推送基础设施选型优化

## 用户故事

- 作为前端接入方，我希望能在 agent 运行过程中逐步看到回复与关键事件，而不是只能等待最终结果。
- 作为后端实现者，我希望流式能力建立在已有 `020` 后端入口之上，不重做 topic/session 胶水。

## 输入输出

- 输入：
  - `topic_id`
  - `user_input`
  - 可选 `attachments`
  - 可选 `metadata`
- 输出：
  - 一条有序的流式事件序列
  - 以及与同步接口兼容的最终完成态结果

## 约束

- `022` 依赖 `020-backend-glue-minimal`
- 不改变 `topic_id -> active_session_id` 的既有映射策略
- 不移除同步 `POST /runs`
- 流式协议只服务主栏与运行过程可见事件
- 第一版流式不要求覆盖完整 topic 真相层更新
- 流式实现必须有明确的完成、失败和关闭语义
- 流式事件必须能被前端稳定消费和回放最小状态

## 流式事件边界

第一版至少应定义以下事件类型：

- `run_started`
- `assistant_delta`
- `tool_call_started`
- `tool_call_finished`
- `run_completed`
- `run_failed`

允许后续扩展更多事件，但首版至少要保证：

- 有开始事件
- 有文本增量事件
- 有工具调用关键事件
- 有完成或失败终态事件

## 与同步接口的关系

- 同步 `POST /runs` 继续保留，作为简单调用路径
- 流式接口作为并行能力新增
- 同一后端服务应同时支持同步与流式两种 run 方式
- 流式完成后，最终状态应与同步 `RunResponse` 的核心结果保持一致

## 前端消费边界

- 前端主栏可以基于流式事件渐进展示 agent 回复
- 工具调用可显示为运行中摘要
- 流式结束后，前端可选择再拉取一次 `GET /messages` 或使用完成态结果做最终对齐

## 传输方式

- 首版允许在实现阶段选择最薄、最常规的流式传输方式
- specs 当前不把具体传输框架或实现库写死
- 但必须满足：
  - 浏览器端可消费
  - 单次 run 的事件顺序稳定
  - 失败与关闭语义清晰

## 验收标准

- 流式运行建立在 `020` 之上，而不是重做后端入口
- 同步与流式 run 能并存
- 前端能逐步收到 agent 回复内容
- 前端能收到关键工具调用事件
- 流式有明确完成态与失败态
- 流式完成后的最终结果可与同步结果对齐
