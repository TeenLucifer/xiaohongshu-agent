# 003 Content Creation

## 背景

在新版 workspace-first 工作台里，模式总结、文案结果和图片结果都收进同一个大工作区。主内容区现在收口为 `创作 / 对话` 两个 tab：搜索结果、总结、文案和图片都留在 `创作`，对话记录和结果消息进入 `对话` tab。

## 目标

- 在 `对话` tab 中以 agent 结果消息展示模式总结和文案结果
- 让文案以对话节点形式进入 `对话` tab，但正式编辑入口固定在 `创作` tab
- 在工作区中展示 Markdown 化的总结、可编辑的 Markdown 文案区和图片结果
- 在右侧文案区提供选区级 AI 润色与内部滚动编辑体验
- 在“图片”section 中同时承接编辑区、真实生成结果展示与局部输入框
- 让上传的图片素材进入“图片”section 的候选图片列表，并排在帖子图之前
- 提供右侧“总结/文案”生成按钮，复用现有 run 链路
- 让“创作”tab 中的图片编辑区与大图预览交互各自有独立能力边界

## 非目标

- 文案多版本对比
- 图片精修画布
- 复杂富文本编辑器

## 用户故事

- 作为运营人员，我希望中间主栏专注看 Agent 对话和结果节点，而不是再看到一堆独立大区块。
- 作为运营人员，我希望文案主要在右侧白板式编辑区中连续编辑，并能对选中的一小段文字单独发起 AI 润色。
- 作为开发者，我希望先固定对话节点和右侧工作区的分工，再接真实生成结果。

## 输入输出

- 输入：真实 `pattern_summary.json`、真实 `copy_draft.json`、真实 `editor_images.json`、真实 `image_results.json`
- 输出：可查看总结、可编辑文案、可浏览右侧图片结果并消费聊天缩略图的三栏工作台状态

## 约束

- 中间主栏保持简化对话流优先
- 文案以 agent 结果消息形式出现，不常驻中间大编辑区
- 主界面不展示独立的结果摘要块
  - 总结结果不做摘要面板
  - 文案结果不做摘要面板
  - 都以 agent 回复消息直接呈现
- 工作区采用两 tab 结构
  - “创作” tab 包含搜索结果、总结、文案、图片
  - “对话” tab 包含 user / agent 对话结果
- “创作” tab 顶部的搜索结果区下方内嵌总结块，采用只读 Markdown 展示并提供固定动作按钮
  - 总结内容区域采用白板式直出，避免多层嵌套边框容器
- “创作” tab 包含文案和图片两个分组，默认折叠
- “创作” tab 中的文案区采用可编辑的单文档 Markdown 展示，提供固定动作按钮
- 文案区编辑交互固定为：
  - 右侧只有一个连续的白板式文档编辑区
  - 保留 `文案` section 头和动作按钮，但正文区域不再套额外内层卡片或说明框
  - 输入 `#` / `##` / `-` / `>` 等 Markdown 语法后，当前块即时切换对应格式
  - 标题与正文编辑都需要自动写回 `copy_draft.json`
  - 正文超出当前显示高度时，由编辑器自身滚动承接长文输入
  - 输入、换行、替换选区后，需要自动将当前光标或选区滚动进可视区
  - 选中文本后可触发局部 AI 润色
  - AI 润色通过弹层输入用户提示词，读取整篇文案上下文，但只返回并替换选区文本
- “创作” tab 中的图片 section 由两部分组成：
  - 上半部分为 `007-image-editor-frontend` 承接的图片编辑区
  - 下半部分为 `027-agent-skills-suite` 中 `image-generation` 产出的图片结果展示区
- 图片编辑区的候选素材来源为两路合并：
  - `028-workspace-materials-pool` 中上传的本地图片
  - 搜索结果帖子图片
  - 顺序固定为上传图片优先、帖子图其次
- “创作” tab 中的图片 section 提供局部输入框，可直接向 agent 发送消息，记录同步进入 `对话` tab
  - 该输入框位于图片编辑区之后、生成结果区之前
  - 输入框采用单层轻量样式，placeholder 固定为 `输入你想继续让agent处理的内容...`
  - 输入框为可自动换行的多行输入区，高度自动增长到上限后内部滚动
- 图片结果区与图片编辑区中的大图预览统一由 `008-image-lightbox-preview` 承接
- 按钮本质上只是代用户发起一次 run，不新增第二套后端动作
- 在 `创作` tab 中点击这些按钮后，页面保持停留在当前 tab
- 按钮无论当前是否已有结果都保持可见，可用于重新生成和覆盖更新
- 图片结果区按单列表追加展示，不再按文生图和图生图分组
- 每张结果图都需要支持删除和拖拽进入编辑区
- final answer 下需支持展示 1 张代表图缩略结果
- 文案节点和右侧文案区都需要采用统一字号层级和按钮样式
- 选区润色完成后，中间对话流需要追加一条执行结果消息
- 右侧文案摘要和图片区需要与帖子结果区保持一致的圆角、边框和间距体系
- `对话` tab 中的结果消息需要服从主阅读列节奏
  - 总结结果和文案结果应作为精简的 agent 回复内容出现
  - `创作` 中的搜索结果、总结、文案、图片区仍保持工作区式的辅助表达

## 验收标准

- 中间对话流能以 agent 回复消息展示模式总结结果
- 文案节点只读展示，提示用户到右侧继续编辑
- 右侧文案区支持单文档 Markdown 白板式编辑，并在内容超长时使用内部滚动
- 右侧文案区编辑后会真实写回 `copy_draft.json`
- 右侧文案区选中文本后可触发 AI 润色，并在成功后直接替换原选区
- 选区润色完成后，中间对话流会新增一条结果消息
- 工作区包含”创作”和”对话”两个 tab
- “创作” tab 顶部能看到搜索结果，以及内嵌在搜索结果区下方的总结块
- “创作” tab 中在搜索结果下方能看到文案和图片，并在图片区看到局部输入框
- 文案区只保留 section 头和单层白板正文，不再出现明显多层嵌套框
- “创作” tab 中各 section 都有固定动作按钮
- 图片结果在“创作” tab 的图片 section 下半部分展示
- 图片结果按单列表追加展示
- 上传本地图片会出现在“创作”tab 的图片区候选列表中，且优先于帖子图
- 搜索结果区和图片区的局部输入发送后，页面停留在当前 tab，且 `对话` tab 可看到对应记录
- final answer 下可展示 1 张代表图缩略结果
- 右侧文案摘要和图片区与帖子区风格一致
- 文案摘要和图片区在更宽右栏下仍不抢中栏焦点
- 中栏总结结果和文案结果符合阅读列节奏，不显得过厚或过满
- 中栏不存在独立结果摘要块，结果内容直接以 agent 回复消息呈现
