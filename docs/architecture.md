# 架构概览

## 目标

本模板采用单仓、单进程、模块边界清晰的 Python AI 工程骨架，目标是让新项目从第一天起就能按 `SDD + TDD` 稳定推进，而不是先堆脚本再回头重构。

## 分层约定

- 交互层：页面、CLI、接口入口与会话状态
- 用例编排层：组织业务主流程，不承担底层适配
- 结构化类型层：定义跨层输入输出和公共数据契约
- 外部能力适配层：封装模型调用、第三方服务或系统接口
- 持久化层：本地文件、数据库、缓存或外部存储适配
- 配置层：环境变量、运行时参数与配置加载
- 通用辅助层：只放通用工具，不承载核心业务规则

## 默认数据流

1. 交互入口接收用户输入或脚本参数
2. 用例编排层组合业务步骤并维护主流程
3. 结构化类型层约束跨层数据形状
4. 如需模型调用，统一经过外部能力适配层
5. 如需持久化或外部资源，统一经过持久化层
6. 交互层负责把结构化结果渲染为用户可见输出

## 设计原则

- 页面或 CLI 不直接拼业务逻辑
- 用例编排层不绕过结构化类型直接读裸 `dict`
- 外部接口和本地持久化都应隔离在适配层
- spec 先行，测试先行，最后再补实现

## 扩展方式

- 新增业务能力时，先在 `specs/` 新建一个 feature
- 若某类能力会长期存在，再补对应模块目录和最小公共接口
- 若项目后续需要框架或工作流引擎，应在 `docs/tech-decisions.md` 中记录决策理由
